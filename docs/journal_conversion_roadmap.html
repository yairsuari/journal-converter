Journal Conversion Tool

Open-source academic manuscript reformatter — Development Roadmap

Working Document · June 2026

Development Phases

Phase 1 Core Conversion Engine Complete
  • Pandoc integration for docx ↔ docx, docx ↔ tex, tex ↔ tex
  • Citation extraction from Paperpile-linked Word docs via BibTeX export
  • CSL-based citation reformatting (author-year ↔ numbered ↔ footnote)
  • Reference list rebuilding using target journal's CSL style
  • Basic journal config schema (YAML): citation style, template path, abstract type
First target: convert between the two specific journals that prompted this project. Everything else builds on this working baseline.
Phase 2 Journal Config System Complete
  • Finalise YAML schema for a journal profile (citation style, Word template, abstract format, structural rules, word limit)
  • Add publisher and last_verified fields to the schema
  • Publisher-level base configs: shared template and rules inherited by all journals under the same publisher (e.g. all Frontiers journals, all Elsevier journals)
  • Journal configs can override publisher defaults for journal-specific rules only
  • Write initial configs for the two seed journals
  • Config validation (missing fields, unresolved CSL names, broken template paths)
  • Establish journals/ folder as the community contribution point
Journal configs are the main extension point — adding support for a new journal should require no core code changes.
Phase 3 CLI Interface Complete
  • Click-based CLI: convert.py source.docx --from journal_a --to journal_b
  • Optional --bib flag to supply a separate BibTeX file
  • First-run configuration wizard (locates Pandoc, sets default template directory)
  • Output: converted document + conversion report (what changed, what needs manual review)
Phase 4 Streamlit GUI Complete
  • File upload: source document + optional .bib file
  • Journal selector dropdown (populated from available configs)
  • Conversion preview / diff before download
  • Download converted document
  • Hosted on Streamlit Community Cloud — no installation required for end users
GUI and CLI are thin wrappers over the same core. Adding the GUI should not require changes to Phase 1–2 code.
Phase 5 Community & Distribution
  • GitHub repository with MIT license
  • Structured issue template for bug reports (source journal, target journal, conversion type, what broke, anonymised example)
  • GitHub Discussions for journal pair requests and feature discussion
  • Contributing guide: how to add a new journal config via pull request
  • README with supported journals table and quick-start instructions
  • Disclaimer (see below)

Resolved Decisions

Hosting
Streamlit Community Cloud (free, zero-friction) vs. self-hosted? Community Cloud has branding and resource limits; self-hosting gives control but adds maintenance burden.
Decision Streamlit Community Cloud. Zero installation for end users — share a URL, done. Acceptable trade-off at this stage; can migrate to self-hosted later if scale demands it.
Structural Differences
How to handle journals with structural divergence: structured vs. unstructured abstract, figures at end of document, highlights section, graphical abstract?
Decision Structural rules are declared in each journal's YAML config. If a required transformation has no config entry, the tool flags it clearly as unsupported rather than silently producing wrong output. For unsupported pairs, the tool will suggest a step-by-step workflow for creating or extending a journal YAML — making it a community contribution opportunity rather than a dead end.

Open Questions

Privacy
How should users anonymise sample manuscripts when submitting bug reports? Need clear guidance — a checklist or a strip-metadata script — to protect authors and avoid IP issues.
Decision No stripping scripts or tooling. Anonymity guidance is included directly in the bug report template: replace author names with placeholders, attach only the relevant excerpt (not the full manuscript), and redact any unpublished references. Responsibility sits with the user; the disclaimer reinforces this.
CSL Versioning
Pin CSL files to a specific version in the repo, or pull from the CSL registry at runtime? Pinning is reproducible but can go stale; live fetching stays current but breaks offline use.
Decision Both: attempt to fetch the latest CSL file from the registry at runtime. If the fetch fails (offline, registry unavailable), fall back to the pinned version bundled in the repo and notify the user. If no pinned version exists either, prompt the user to supply the CSL file manually.
Plain-text Citations
Manuscripts with no reference manager (citations typed as plain text) cannot be automatically reformatted. Out of scope for v1, or best-effort detection with a warning?
Decision Out of scope for v1. The tool will detect the likely absence of structured citations and show a clear warning. If the user supplies a .bib file separately, the tool will attempt a best-effort match — no guarantees, but it won't silently fail.
LaTeX Templates
Journal .cls files: bundle in repo (licensing risk?) or require the user to supply them? Many journals distribute their own class files freely but retain copyright.
Decision Check the license declared in the .cls file header. Files with LPPL, MIT, or equivalent permissive licenses are bundled in the repo. Files with no license or a restrictive one are not bundled — the tool prompts the user to place the journal's .cls file in the expected folder.
Config Versioning
Journal submission guidelines change. Should configs carry a version/date field so users know when a config was last verified against the journal's current requirements?
Decision Configs carry a last_verified date field so users can judge staleness. Configs also declare a publisher field — many publishers (e.g. Elsevier, Frontiers, Wiley) apply near-identical templates across journals, enabling config reuse and cross-journal consistency checks.
Bug Report Location
GitHub Issues is the natural fit, but some academics are not comfortable with GitHub. Worth adding a fallback (Google Form that creates an issue automatically)?
Decision GitHub Issues as the primary channel. A Google Form is provided as a no-account fallback — submissions are forwarded to the maintainer who files the issue. Lowers the barrier for academics unfamiliar with GitHub.

Planned Technology Stack

Layer Technology Role
Document conversion Pandoc Backbone for all format conversions; handles internal document model
Python bindings pypandoc, python-docx Drive Pandoc from Python; inspect/modify Word document structure
Citation processing Pandoc citeproc + CSL files Reformat inline citations and reference lists to any target style
Citation source BibTeX (exported from Paperpile) Structured metadata for all references; feeds citeproc
Journal configs YAML Per-journal profiles; community-contributed via pull requests
CLI Click Command-line interface and first-run wizard
GUI Streamlit Browser-based interface; hostable on Streamlit Community Cloud
License MIT Permissive; compatible with all planned dependencies
Planned Disclaimer (draft)

This tool performs automated reformatting and does not guarantee compliance with a journal's current submission requirements. Output must be reviewed manually before submission. The maintainers accept no responsibility for manuscripts rejected due to formatting issues. Supported journal configurations are community-maintained and may not reflect the most recent author guidelines. Always verify against the target journal's current instructions for authors.
